# Contributing to librechat-prom-exporter

Thanks for considering a contribution! Detailed guides live on the docs site:

- **[Ways to contribute](https://rubentalstra.github.io/librechat-prom-exporter/docs/contributing/ways-to-contribute)** — bugs, metrics, dashboards, docs, sponsorship.
- **[Architecture](https://rubentalstra.github.io/librechat-prom-exporter/docs/contributing/architecture)** — why the code is shaped the way it is. Read this before non-trivial changes.
- **[Adding a metric](https://rubentalstra.github.io/librechat-prom-exporter/docs/contributing/adding-a-metric)** — step-by-step playbook with cardinality budget rules.

## Ground rules

- All contributions are MIT-licensed (see [LICENSE](LICENSE)).
- Be excellent — see [CODE_OF_CONDUCT.md](CODE_OF_CONDUCT.md).
- **Security issues** — please use the [private security advisory flow](https://github.com/rubentalstra/librechat-prom-exporter/security/advisories/new), not a public issue. See [SECURITY.md](SECURITY.md).
- **Disclose AI usage** — if a contribution was substantially generated by an AI assistant, mention it in the PR description.

## Cutting a release (maintainers)

The release is mostly automated. Two steps:

1. **Prepare**: Actions tab → **"Prepare release"** workflow → Run workflow → enter the new version (e.g. `0.9.0`). The workflow snapshots the docs site, bumps `package.json`, and opens a PR titled "Release v0.9.0". Review the diff, merge it.

2. **Tag**: after the release PR merges, tag the merge commit:

   ```bash
   git fetch origin main
   git tag v0.9.0 origin/main
   git push origin v0.9.0
   ```

   Or use the GitHub UI: **Releases → Draft a new release → tag `v0.9.0` on `main` → Generate release notes → Publish**.

CD takes over from there: multi-arch build, cosign sign, attest build provenance + SBOM, Trivy scan, attach SBOM to the GitHub Release. The docs site auto-deploys when the merge lands.

Why two steps? GitHub's branch protection requires all changes to `main` to go through a PR. Splitting "prepare PR" from "tag" lets the prep PR get the same CI + review as any other change, and keeps the tag on the actual merge commit.
